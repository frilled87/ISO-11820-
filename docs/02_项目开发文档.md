## 1 项目概述

### 1.1 项目背景

ISO 11820:2022 是国际标准化组织发布的建筑材料不燃性试验标准，规定了将建筑材料样品置于 750°C 加热炉中、连续记录 60 分钟多通道温度数据、依据温升和失重率判定材料是否合格的完整试验流程。

在高校实验教学场景中，单台 ISO 11820 试验炉造价 20-50 万元，需配套 PID 温控系统、Modbus 采集模块等硬件，学生人均操作机会严重不足，且单次实验耗时 60 分钟以上，教学效率低下。当前市场上缺乏面向教学场景的开源仿真工具，学生无法在课前预习、课后复现实验流程。

本系统以"零硬件依赖的全流程仿真"为核心设计理念，采用 Java 17 + JavaFX 21 技术栈，开发一套完整的 ISO 11820 建筑材料不燃性试验仿真桌面应用，覆盖从登录、新建试验、升温仿真、数据记录、现象录入到 CSV/Excel/PDF 多格式报告导出的全流程，为高校实验教学提供低成本、高效率的数字化解决方案。

### 1.2 项目定位

#### 1.2.1 应用场景

| 场景 | 说明 |
|------|------|
| 高校实验教学 | 建筑材料/土木工程专业课程实验，支持课堂即时演示和学生自主操作 |
| 课前预习与课后复现 | 学生个人电脑安装运行，无硬件依赖，可反复练习实验流程 |
| 检测机构新人培训 | 建材检测员上岗前标准化流程培训，降低真实设备操作风险 |
| 标准宣贯与演示 | ISO 11820 标准流程可视化演示，用于学术交流和行业培训 |

#### 1.2.2 目标人群

| 用户 | 角色 | 核心需求 |
|------|------|---------|

### 1.3 项目方案

本系统采用**分层模块化架构**，由 5 位组员各自负责独立模块，通过 Git 分目录协同开发。系统整体分为四层：

- **UI 层（JavaFX 21 + FXML）**：登录窗口、主控制面板、实时温度曲线、业务弹窗
- **业务核心层（Core Engine）**：试验状态机、多阶段温度仿真引擎、试验主调度控制器
- **服务层（Service Layer）**：配置管理、CSV 数据读写、Excel/PDF 报告生成、日志系统
- **数据层（Data Layer）**：SQLite 数据库、泛型 BaseDao<T> CRUD 封装、实体映射

数据通过**事件机制（DataChangeListener）**从下层向上层传递，各层之间接口清晰、依赖单向。

### 1.5 项目价值

- **教学价值**：将 60 分钟物理实验压缩为可配置时长的仿真实验，教学效率提升 10 倍以上
- **经济价值**：零硬件成本，相比商业方案（含硬件 20-50 万元）实现零成本部署
- **技术价值**：探索 AI 辅助全流程软件开发（Claude Code）的实践模式，为软件工程教学提供新范式
- **开源价值**：填补 ISO 11820 仿真软件开源生态空白，计划赛后开源

## 5 概要设计

### 5.1 处理流程

```
用户登录 → 主界面 → 新建试验 → 开始升温
                                    ↓
          仿真引擎每 800ms 迭代 ← 状态机 PREPARING
                                    ↓
          温度达到 745~755°C 且稳定 → 状态机 READY
                                    ↓
          用户点击开始记录 → 状态机 RECORDING
                                    ↓
          每秒记录一行温度数据 → 实时更新曲线和统计
                                    ↓
          到达目标时长 / 手动停止 → 状态机 COMPLETE
                                    ↓
          填写试验现象 → 保存到数据库 → 导出报告
```

#### 5.8.1 错误/异常输出信息

| 异常类型 | 处理策略 | 输出 |
|---------|---------|------|
| 密码错误 | 显示错误提示，不清空输入框 | UI Label "密码错误，请重新输入" |
| 状态非法切换 | 返回 false，记录 WARNING 日志 | 日志 "状态切换拒绝：不允许从 X 切换到 Y" |
| 数据库连接失败 | 抛出 DaoException，包含原始 SQLException | 日志 + 弹窗提示 |

#### 5.8.2 错误/异常处理对策

| 对策 | 说明 |
|------|------|
| 防御式编程 | 所有 public 方法入参非空校验，`Objects.requireNonNull()` |
| 日志记录 | 所有异常通过 SLF4J 记录到文件，含堆栈信息 |
| 优雅降级 | 配置读取失败使用默认值，文件不存在返回空列表不崩溃 |
| 用户提示 | 关键错误通过 Alert 弹窗告知用户，非关键错误仅日志记录 |
| 事务回滚 | 数据库批量操作使用事务，失败自动回滚 |
| 线程安全 | 状态机用 AtomicReference、调度器用 tickLock、数据缓存用 CopyOnWriteArrayList |

## 8 详细设计

### 8.1 核心引擎模块（组员1）

#### 8.1.1 功能描述

核心引擎模块是系统的业务逻辑中枢，负责试验状态管理、温度仿真数据生成、试验主调度控制。该模块由 8 个 Java 源文件组成，代码量 3,026 行，是系统最核心的业务模块。

#### 8.1.2 性能描述

- 仿真迭代周期：800ms（固定，由 ScheduledExecutorService 调度）
- 状态切换延迟：< 1ms（AtomicReference 原子操作）
- 监听器通知：同步执行，单个监听器异常不影响其他监听器
- 温漂计算：O(n) 线性回归，n ≤ 600 数据点

#### 8.1.3 输入

| 输入 | 来源 | 说明 |
|------|------|------|
| 试验状态变更指令 | UI 层 | startHeating/startRecording/stopRecording/stopHeating |
| 仿真参数配置 | AppConfig | 目标温度、升温速率、噪声系数、稳定阈值 |
| 传感器配置 | DAO 层 | 5 通道温度量程和零点 |

#### 8.1.4 输出

| 输出 | 去向 | 说明 |
|------|------|------|
| 5 通道温度数据 | DataChangeListener → UI | 每 800ms 推送一次 |
| 试验状态 | DataChangeListener → UI | 状态变更时推送 |
| 系统消息 | DataChangeListener → UI | 状态变更/异常时推送 |
| 时序温度数据 | CsvDataService | 记录阶段每秒追加 |
| 试验统计结果 | DAO 层 | 试验完成时保存到 testmaster 表 |

#### 8.1.5 程序逻辑

**TestStateMachine（状态机）**：
1. 使用 `AtomicReference<TestState>` 存储当前状态
2. `transitionTo(target)` 校验 `canTransitionTo(target)` 后原子更新
3. 锁外通知所有监听器（不可变副本遍历，防止并发修改）
4. `checkAndHandleReadiness(tf1)` 在 READY 状态检测温度跌出范围自动回退

**SensorSimulator（仿真引擎）**：
1. `update()` 方法根据当前状态选择算法分支（IDLE/PREPARING/READY/RECORDING/COMPLETE）
2. 升温阶段：`TF1 += heatingRate × 0.8 + random(-fluctuation, +fluctuation)`
3. 稳定阶段：`TF1 = targetTemp + random(-fluctuation, +fluctuation)`
4. 记录阶段：炉温维持稳定，表面温/中心温指数逼近各自目标
5. 降温阶段：`TF1 -= 0.5 + noise × 0.1`

**TestMaster（主控制器）**：
1. `startHeating()` 创建单线程调度器，每 800ms 执行 `onTick()`
2. `onTick()` 调用 `simulator.update()` → 根据状态执行对应业务逻辑 → 锁外通知监听器
3. `handlePreparingTick()` 检测 `simulator.isStable()` 自动切换到 READY
4. `handleRecordingTick()` 每秒追加数据、更新统计、检测终止条件
5. `stopRecording()` 时长不足 30 秒自动退回 READY

#### 8.1.6 限制条件

- 仅支持单炉仿真（单 TestMaster 实例）
- 仿真模式与硬件模式通过配置开关隔离，但硬件模式代码未实现
- 调度器为单线程，不支持多炉并行

### 8.2 数据持久层模块（组员2）

#### 8.2.1 功能描述

数据持久层模块负责 SQLite 数据库的完整生命周期管理，包括建表初始化、泛型 CRUD 封装、实体映射、事务管理和连接池。该模块由 20+ 个 Java 源文件组成，代码量 3,327 行。

#### 8.2.2 性能描述

- 单一查询响应时间：< 50ms（SQLite 本地文件）
- 批量操作：使用 PreparedStatement addBatch，支持 1000+ 行/秒
- 字段映射：ConcurrentHashMap 缓存反射结果，首次查询后零反射开销
- 事务：支持手动 begin/commit/rollback

#### 8.2.3 输入

| 输入 | 来源 | 说明 |
|------|------|------|
| SQL 语句 + 参数 | Core/UI 层 | 通过 DAO 接口调用 |
| 建表脚本 | schema.sql | 首次运行时读取并执行 |

#### 8.2.4 输出

| 输出 | 去向 | 说明 |
|------|------|------|
| 实体对象 | Core/UI 层 | 查询结果映射为 Java 实体 |
| 受影响行数 | Core/UI 层 | INSERT/UPDATE/DELETE 返回值 |
| 数据库文件 | 文件系统 | Data/ISO11820.db |

#### 8.2.5 程序逻辑

**BaseDao<T>（泛型 CRUD 基类）**：
1. 通过反射解析泛型参数获取实体类型 T
2. `executeQuery(sql, params)` 使用 PreparedStatement 防注入
3. `mapResultSet(rs)` 遍历 ResultSetMetaData，列名 → 字段名反射赋值
4. 字段名匹配支持三种命名约定：精确匹配 → 首字母小写 → 下划线转驼峰
5. 字段缓存使用 ConcurrentHashMap，首次初始化后零反射开销

**DbUtil（连接管理）**：
1. 首次调用 `getConnection()` 自动执行 `initDatabase()` 幂等建表
2. 支持 ThreadLocal 连接绑定（批量操作复用）
3. 支持事务：`beginTransaction()` → `commitTransaction()` / `rollbackTransaction()`
4. 数据库版本管理：`system_config` 表记录 db_version，支持逐级升级

#### 8.2.6 限制条件

- 仅支持 SQLite 数据库引擎
- 列名与字段名必须完全一致（或遵循命名转换规则）
- 实体类必须有无参构造方法
- DAO 测试必须使用临时数据库，不污染正式数据

### 9.3 覆盖率门禁

```xml
<!-- pom.xml JaCoCo 配置 -->
<rule>
    <element>BUNDLE</element>
    <limits>
        <limit>
            <counter>INSTRUCTION</counter>
            <value>COVEREDRATIO</value>
            <minimum>0.80</minimum>
        </limit>
    </limits>
</rule>
```

**执行流程**：
1. 组员1-4 提交代码到各自 Git 目录
2. 组员5 编写/更新对应测试用例
3. 执行 `mvn clean verify` 运行全量测试 + 覆盖率检查
4. 不达标 → 构建失败 → 通知对应组员补充测试
5. 达标 → 生成 JaCoCo 报告（`target/site/jacoco/index.html`）

### 9.4 测试隔离机制

| 隔离项 | 方案 |
|--------|------|
| 数据库隔离 | 每个测试方法使用 `Files.createTempFile()` 创建独立临时 SQLite 数据库，@AfterEach 自动删除 |
| 仿真可重现 | SensorSimulator 使用固定 Random seed（42），确保测试结果可重现 |
| 时间依赖 | 系统消息时间戳使用 `LocalTime.now()`，测试不验证具体时间值 |
| 文件隔离 | 工具类测试使用 `@TempDir` 注解创建临时目录 |
