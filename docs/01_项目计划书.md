## 1 项目概述

### 1.1 项目背景

#### 1.1.1 行业痛点

建筑材料防火性能检测是建筑安全领域的核心环节。ISO 11820:2022 国际标准规定了建筑材料不燃性试验的完整流程：将样品置于 750°C 加热炉中，连续记录 60 分钟内的多通道温度数据，依据温升和失重率判定材料是否合格。

然而，当前国内高校和建材检测机构的实验教学面临三大困境：

| 痛点 | 现状 | 影响 |
|------|------|------|
| **硬件依赖高** | 单台 ISO 11820 试验炉造价 20-50 万元，需配套 PID 温控系统、Modbus 采集模块、串口通信硬件 | 高校实验室难以批量配置，学生人均操作机会不足 |
| **实验周期长** | 单次完整试验需 60 分钟升温 + 记录，日均仅能完成 3-4 组 | 教学效率低，无法满足全班实操需求 |
| **软件生态缺失** | 商业试验软件绑定特定硬件，开源替代品空白，教学场景无仿真工具可用 | 学生无法在课前预习、课后复现实验流程 |

#### 1.1.2 项目价值

本系统以 **"无硬件依赖的全流程仿真"** 为核心设计理念，开发一套面向高校实验教学的 ISO 11820 建筑材料不燃性试验仿真系统，实现：

- **零硬件成本**：纯软件仿真，学生个人电脑即可运行完整试验流程
- **教学提效**：从 60 分钟物理实验缩短为可配置时长的仿真实验，支持课堂即时演示
- **标准化输出**：自动生成 CSV/Excel/PDF 三种格式试验报告，符合 ISO 11820 标准格式
- **AI 协作开发**：全组使用 Claude Code 辅助编码与文档编写，探索 AI 时代软件工程实践新模式

### 1.2 项目内容

#### 1.2.1 系统定位

| 开发语言 | Java 17 |

#### 1.2.2 核心功能

本系统完整覆盖 ISO 11820:2022 标准试验的全部操作流程：

```
登录 → 新建试验 → 开始升温 → 温度稳定 → 开始记录 → 试验完成 → 导出报告
```

具体包括：

1. **角色登录系统**：管理员/试验员双角色，密码验证
2. **试验信息管理**：样品信息录入、环境参数配置、试验时长模式选择
3. **多阶段温度仿真引擎**：5 通道（炉温1、炉温2、表面温、中心温、校准温），严格按 ISO 11820 算法生成升温/稳定/记录/降温四阶段温度数据
4. **试验状态机**：5 状态（IDLE→PREPARING→READY→RECORDING→COMPLETE）自动流转，含温度稳定性自动判定和异常回退逻辑
5. **实时数据可视化**：XChart 温度曲线图、LED 风格数值显示、计时器、温漂监测
6. **试验现象记录**：火焰发生时刻与持续时间、试验后质量、失重率自动计算
7. **多格式报告导出**：CSV 时序数据、Excel 含图表报告、PDF 正式试验报告
8. **历史记录查询**：按日期/样品/操作员多条件筛选，支持分页和详情查看
9. **设备校准管理**：传感器标定数据记录与历史查询

### 1.3 创新点

#### 1.3.1 技术创新点

| 创新点 | 描述 | 技术实现 |
|--------|------|---------|
| **多阶段温度仿真算法** | 严格依据 ISO 11820 标准，实现升温/稳定/记录/降温四阶段独立算法，支持可配置的升温速率、噪声系数、稳定阈值 | SensorSimulator 类，每 800ms 迭代一次，固定种子可重现 |
| **有限状态机驱动** | 5 状态试验控制器，含自动温度判定、异常回退、保存状态保护逻辑 | TestStateMachine + TestMaster，AtomicReference 线程安全 |
| **指数逼近表面/中心温模型** | 记录阶段表面温向炉温×0.95 指数接近，中心温向炉温×0.85 指数接近（比表面温慢一半），模拟真实热传导 | 一阶指数平滑算法，系数 0.02/0.01 |
| **温漂最小二乘线性回归** | 对最近 10 分钟炉温序列做最小二乘拟合，计算温漂斜率，作为提前终止条件判定依据 | calcTempDrift() 方法，O(n) 等差数列求和优化 |
| **内存数据库隔离测试** | DAO 层测试使用临时 SQLite 文件，测试后自动删除，不污染正式数据 | BaseDaoTest 基类，@BeforeEach/@AfterEach 自动管理 |

## 2 技术方案

### 2.1 总体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        UI 层 (JavaFX 21)                         │
│  LoginController │ MainController │ NewTestDialog │ TestRecord  │
│  LoginView.fxml  │  MainView.fxml  │  XChart 曲线图  │  LED 面板  │
├─────────────────────────────────────────────────────────────────┤
│                    业务核心层 (Core Engine)                       │
│  TestStateMachine │ TestMaster │ SensorSimulator │ TestContext  │
│  状态机(5状态)    │ 主调度器    │ 仿真引擎(800ms) │ 试验上下文    │
├─────────────────────────────────────────────────────────────────┤
│                    服务层 (Service Layer)                         │
│  CsvDataService │ ExcelReportService │ PdfReportService          │
│  ReportExportService │ AppConfig │ LogUtil │ FilePathManageUtil │
├─────────────────────────────────────────────────────────────────┤
│                    数据层 (Data Layer)                            │
│  BaseDao<T> │ OperatorDao │ TestMasterDao │ DbUtil (SQLite)     │
│  Entity 层: Operators, TestMaster, ProductMaster, Sensors ...   │
├─────────────────────────────────────────────────────────────────┤
│                    基础设施                                        │
│  SQLite 3 │ Maven │ JUnit 5 │ JaCoCo │ logback │ Jackson │ iText │
└─────────────────────────────────────────────────────────────────┘
```

**分层原则**：

- 上层依赖下层，下层**不感知**上层
- 数据向上传递通过**事件机制**（DataChangeListener），不直接调用 UI 方法
- 全局单例（AppContext）持有核心对象引用，避免全局变量污染
- 所有 UI 更新必须在 UI 线程执行（JavaFX `Platform.runLater()`）

#### 2.2.2 试验状态机流转

```
IDLE（空闲）
  └─ 用户点击「开始升温」→ PREPARING（升温中）
      ├─ 温度达到 745~755°C 且稳定 → READY（就绪）
      │   ├─ 用户点击「开始记录」→ RECORDING（记录中）
      │   │   ├─ 到达目标时长 → COMPLETE（完成）
      │   │   └─ 时长不足 30 秒 → READY（回退）
      │   ├─ 温度跌出稳定范围 → PREPARING（自动回退）
      │   └─ 用户点击「停止加热」→ IDLE
      └─ 用户点击「停止加热」→ IDLE
```

## 4 总结与展望

### 4.1 可行性分析总结

本项目的可行性已从技术、资源、市场三个维度得到充分验证：

1. **技术可行**：全部技术选型为业界成熟方案，核心技术难点（仿真算法、状态机、内存测试、三格式报告）均已完成代码实现并通过单元测试验证
2. **资源可行**：5 人模块化分工模式运转良好，代码已交付 16,000+ 行，测试覆盖 140+ 用例，所有模块均按计划完成
3. **市场可行**：面向高校实验教学场景，填补了开源 ISO 11820 仿真软件的空白，相比商业方案具有零成本、可定制、教学友好三大优势

### 4.2 项目展望

1. **硬件模式对接**：当前架构已预留 `EnableSimulation` 开关，可通过实现 Modbus 采集接口对接真实试验炉，实现仿真/硬件双模式运行
2. **多炉同时控制**：当前为单炉设计，通过状态机实例化扩展可支持多炉并行试验
3. **云端数据管理**：可扩展后端服务，将试验数据同步至云端，支持多端查看和历史数据统计分析
4. **开源社区建设**：计划将项目开源，吸引建材检测领域开发者贡献代码和测试用例
