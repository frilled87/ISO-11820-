## 1 问题聚焦

### 1.1 问题描述

建筑材料不燃性检测是建筑防火安全的第一道技术关口。ISO 11820:2022 国际标准规定了严格的试验流程：将建筑材料样品置于 750°C 加热炉中，通过多通道热电偶（炉温1、炉温2、表面温、中心温、校准温）连续采集 60 分钟的温度数据，依据温升（deltatf）和失重率（lostweight_per）两项指标判定材料是否合格。

然而，围绕该标准试验的技术生态存在三个结构性问题：

**问题一：行业数字化缺失 — 实验数据管理处于"前信息化"阶段**

当前国内建材检测实验室的数据管理普遍依赖以下模式：试验设备输出原始温度读数 → 检验员手动抄录到纸质记录表 → 人工计算温升和失重率 → 手动编写 Word 报告。该模式存在以下技术缺陷：

| 缺陷 | 技术表现 | 影响 |
|------|---------|------|
| 数据孤岛 | 温度时序数据存储在设备私有内存中，试验结束后无法导出通用格式 | 数据无法跨平台分析、无法长期归档 |
| 计算误差 | 人工计算温升和失重率，有效数字取舍不一致 | 判定结论的精度不可控 |
| 报告非标 | 各实验室报告格式不统一，无标准化模板 | 跨实验室数据不可比 |
| 无版本追溯 | 纸质记录无修改历史，数据篡改无法审计 | 质量管理体系不闭合 |

**问题二：仿真逻辑标准化不足 — 现有仿真工具缺乏 ISO 标准语义**

当高校或检测机构试图使用仿真软件替代真实硬件进行教学或预演时，面临"仿真逻辑与标准流程脱节"的问题。通用仿真工具（如 MATLAB Simulink、COMSOL）可以模拟热传导物理过程，但缺少以下 ISO 11820 标准特有的业务语义：

- 试验状态机（IDLE → PREPARING → READY → RECORDING → COMPLETE）的自动流转逻辑
- 升温阶段到稳定阶段的自动判定条件（745~755°C 且连续稳定 3.2 秒以上）
- 记录阶段表面温/中心温的热传导延迟模型（指数逼近而非线性跟随）
- 温漂提前终止条件（10 分钟温漂 < 2°C/10min 可提前结束试验）
- 试验完成后的保存状态保护（flag = "10000000"）

这些标准语义需要在仿真软件中以**显式的、可验证的算法**实现，而非隐藏在通用物理模型背后。

**问题三：行业软件无自动化质量校验 — 测试体系缺失**

当前建材检测领域的商业软件和开源工具普遍存在"重功能、轻测试"的问题。具体表现为：

- 无单元测试：温度仿真算法的正确性依赖人工目测曲线，无数学验证
- 无回归测试：软件升级后无法自动验证已有功能是否退化
- 无覆盖率度量：代码质量无法量化评估
- 无测试隔离：测试数据与生产数据混合，测试操作可能污染正式数据库

这三个问题共同构成了本项目的技术研究动机：**如何在零硬件成本的前提下，构建一套标准合规、数据完整、质量可控的 ISO 11820 仿真系统？**

### 1.2 问题抽象

将上述三个领域问题抽象为以下技术研究问题：

| 领域问题 | 技术抽象 | 研究问题 |
|---------|---------|---------|
| 行业数字化缺失 | 数据持久化与标准化 | 如何设计一套完整的数据模型，覆盖试验全生命周期的结构化数据（关系型）和非结构化数据（时序）？ |
| 仿真逻辑标准化不足 | 领域算法建模 | 如何将 ISO 11820 标准的自然语言描述转化为可验证的数学算法，并保证算法的可重现性和可配置性？ |
| 行业软件无自动化质量校验 | 软件质量工程 | 如何在学生团队项目中建立工业级的自动化测试体系，实现代码覆盖率 ≥ 80% 的量化质量目标？ |

### 1.3 问题定位

本项目的技术定位是：**面向高校实验教学场景的 ISO 11820 标准合规仿真系统**。该定位在技术难度、工程可行性和应用价值三个维度上具有明确的边界：

| 定位维度 | 包含 | 不包含 |
|---------|------|--------|
| 仿真模式 | 纯软件仿真，算法驱动温度生成 | 真实硬件 Modbus 通信（架构预留接口） |
| 标准覆盖 | ISO 11820:2022 核心试验流程 | 标准全文合规认证 |
| 目标用户 | 高校教师/学生、检测机构培训人员 | 生产环境检测人员 |
| 质量目标 | 核心业务代码覆盖率 ≥ 80% | 100% 全覆盖（UI 层除外） |
| 技术栈 | Java 17 + JavaFX 21 + SQLite + Maven | 云端部署、微服务架构 |

### 1.4 问题评估

对三个核心问题进行技术难度和业务影响评估：

| 问题 | 技术难度 | 业务影响 | 可解性 | 优先级 |
|------|:------:|:------:|:------:|:------:|
| 数据持久化与标准化 | 中 | 高 | 高（SQLite + CSV 双存储方案成熟） | P0 |
| 领域算法建模 | 高 | 高 | 中（需深入理解 ISO 11820 数学语义） | P0 |
| 自动化质量校验 | 中 | 中 | 高（JUnit + JaCoCo 生态成熟） | P1 |

**风险评估**：

- **最高风险**：领域算法建模。ISO 11820 标准文本以自然语言描述试验流程，将其转化为精确的数学公式存在理解偏差的风险。缓解措施：使用固定随机种子确保算法可重现，编写 16 个专项测试用例验证每个算法阶段的数学正确性。
- **中等风险**：数据持久化。SQLite 的并发写入限制可能成为性能瓶颈。缓解措施：使用写锁（ReentrantReadWriteLock）保护 CSV 写入，使用 CopyOnWriteArrayList 缓存时序数据。
- **低风险**：自动化质量校验。JUnit 5 + JaCoCo 是业界成熟方案，集成难度低。

将三个核心问题分解为可独立求解的子问题，并与 5 位组员的模块分工对应：

## 2 相关工作

本章梳理与本项目技术路线相关的三类现有工作：温度仿真工具、Java 桌面开发框架、自动化测试方案。

### 2.1 温度仿真工具

| 工具/方法 | 技术路线 | 优势 | 不足 | 与本项目的关系 |
|----------|---------|------|------|-------------|
| **MATLAB Simulink** | 基于物理模型的热传导偏微分方程求解 | 仿真精度高，可视化能力强 | 需专业建模知识，无 ISO 11820 业务流程，许可费用高 | 参考其热传导指数逼近模型，但采用简化的一阶指数平滑替代 PDE 求解 |
| **COMSOL Multiphysics** | 有限元多物理场耦合仿真 | 可模拟真实几何形状的温度分布 | 计算资源消耗大，实时性差，无标准流程引导 | 参考其多通道独立仿真的思路，但简化为 5 通道独立算法 |
| **Python matplotlib+numpy** | 脚本式温度曲线生成 | 开发快速，适合原型验证 | 无 GUI、无状态管理、无数据持久化 | 作为算法原型验证工具，核心算法验证后迁移到 Java |
| **商业试验炉配套软件** | 硬件绑定，Modbus 协议采集 | 与真实硬件无缝集成 | 闭源，无仿真模式，不可教学化 | 参考其 UI 布局和操作流程设计 |
| **高校课程 Demo** | 简单的线性插值 + 随机数 | 实现简单 | 不符合 ISO 标准，无可重现性，无测试 | 作为对比基线，本项目的算法精度和软件质量均远超该水平 |

**本项目的技术路线选择**：吸取 MATLAB/COMSOL 的物理建模思想（指数逼近），借鉴商业软件的操作流程设计，但采用**简化的算法模型 + 标准化的业务流程**，在仿真精度和工程可行性之间取得平衡。核心创新在于将 ISO 11820 标准文本中的自然语言描述（"升温至 750°C 并稳定"）转化为可验证的数学判定条件（"TF1 ≥ 747°C 且连续 3.2 秒内温度波动 ≤ 3°C"）。

### 2.2 Java 桌面开发框架

| 框架 | 技术路线 | 优势 | 不足 | 选型结论 |
|------|---------|------|------|---------|
| **JavaFX 21 + FXML** | Oracle 开源维护，声明式 UI + CSS 样式分离 | 现代 UI 组件、Scene Builder 可视化编辑、跨平台 | 需单独配置 Maven 插件，社区规模小于 Swing | ✅ **选用** |
| **Swing** | Java 传统 GUI 工具包 | JDK 内置，无需额外依赖 | 组件老旧，布局管理器复杂，不支持 CSS | ❌ 不选用 |
| **SWT (Eclipse)** | 操作系统原生组件 | 外观接近原生应用 | 需平台相关 JAR，打包复杂 | ❌ 不选用 |
| **Electron** | Web 技术栈（HTML/CSS/JS） | 生态丰富，UI 灵活 | 内存占用大，启动慢，不符合 Java 技术栈要求 | ❌ 不选用 |

**JavaFX 选型理由**：
1. FXML 声明式布局支持 Scene Builder 可视化编辑，降低 UI 开发门槛（组员3）
2. CSS 样式分离实现 UI 与逻辑解耦，便于后续换肤
3. `javafx-swing` 模块提供 SwingNode，可嵌入 XChart（Swing 组件）到 JavaFX 场景
4. `javafx-maven-plugin` 支持一键打包为可执行 JAR

### 2.3 自动化测试方案

| 方案 | 技术路线 | 优势 | 不足 | 选型结论 |
|------|---------|------|------|---------|
| **JUnit 5 + JaCoCo** | 单元测试框架 + 字节码覆盖率插桩 | 生态成熟、Maven 集成简单、支持 @Nested/@ParameterizedTest | 无内置 Mock 功能 | ✅ **选用** |
| **TestNG** | 替代 JUnit 的测试框架 | 支持测试组、依赖测试、参数化更灵活 | 社区规模小于 JUnit 5，Maven 插件兼容性差 | ❌ 不选用 |
| **Mockito + JUnit 5** | Mock 框架 + 单元测试 | 可模拟外部依赖 | 本项目模块间依赖简单，Mock 需求少 | ❌ 暂不需要 |
| **Selenium / TestFX** | UI 自动化测试 | 可模拟用户操作 | 配置复杂，JavaFX 支持不完善 | ❌ 后续考虑 |

**JUnit 5 + JaCoCo 选型理由**：
1. `@Nested` 注解支持按功能模块分层组织测试用例（如 `TestStateMachineTest` 中按"合法切换/非法切换/监听器/线程安全"分为 4 层嵌套类）
2. `@ParameterizedTest` + `@CsvSource` 支持参数化测试，减少重复代码
3. `@DisplayName` 支持中文命名，测试报告可读性强
4. JaCoCo 的字节码插桩方式无需修改业务代码，支持 Maven `verify` 阶段自动门禁检查

## 3 技术方案

### 3.1 技术方向

基于问题分解的结果，本项目的技术方向分为三条主线：

**主线一：分层数据持久化架构**

采用"关系型数据库（SQLite）+ 时序文件（CSV）"的混合存储策略。关系型数据（试验记录、样品信息、用户账号）存入 SQLite，利用其 ACID 事务和 SQL 查询能力；时序温度数据（每秒 5 通道 × 3600 秒 = 18,000 个数据点）存入 CSV 文件，利用其顺序写入的高性能和通用可读性。数据访问层采用泛型 `BaseDao<T>` 封装，通过反射实现 ResultSet 到 Entity 的自动映射，消除重复 CRUD 代码。

**主线二：领域驱动的仿真引擎**

将 ISO 11820 标准文本中的试验流程抽象为两个核心组件：

- **TestStateMachine（状态机）**：将 5 个试验状态（IDLE/PREPARING/READY/RECORDING/COMPLETE）建模为枚举，每个状态声明 `canTransitionTo(target)` 规则，实现状态流转规则的自包含。使用 `AtomicReference` 保证线程安全，通过 `Consumer<StateChangeEvent>` 监听器模式向 UI 层通知状态变更。
- **SensorSimulator（仿真引擎）**：将四阶段温度变化建模为独立的数学函数，每 800ms 迭代一次。升温阶段使用线性增长 + 独立噪声；稳定阶段使用目标温度钳位 + 稳定计数器；记录阶段使用一阶指数平滑（表面温系数 0.02、中心温系数 0.01）；降温阶段使用线性衰减 + 钳位下限。

**主线三：自动化质量门禁**

在 Maven 构建生命周期中嵌入 JaCoCo 覆盖率检查。`mvn verify` 阶段自动执行全量单元测试，采集覆盖率数据，与 80% 阈值比较。不达标则构建失败，阻止低质量代码合入。

#### 3.2.2 关键技术选型深度对比

**（1）数据库方案对比**

| 维度 | SQLite | H2 | MySQL |
|------|--------|----|-------|
| 部署复杂度 | 零安装，单文件 | 零安装，可嵌入 | 需安装服务端 |
| 并发支持 | 读并发，写串行 | 读并发，写串行（嵌入模式） | 完整并发 |
| 适用场景 | 桌面应用 | Web 应用测试 | 生产服务端 |
| JDBC 驱动大小 | ~3MB | ~1.5MB | ~2MB |
| 与本项目匹配度 | ✅ 桌面应用，无需并发写入 | ⚠️ 功能重叠 | ❌ 过重 |

**结论**：选择 SQLite，理由为桌面应用无需服务端、零安装部署、单文件便于分发。

**（2）图表库对比**

| 维度 | XChart | JFreeChart |
|------|--------|-----------|
| 实时更新 API | `updateXYSeries()` 简洁 | 需重建 Dataset |
| 依赖大小 | ~500KB | ~2MB |
| Swing 兼容 | 原生 Swing 组件 | 原生 Swing 组件 |
| 文档质量 | 英文，示例丰富 | 英文，文档较旧 |
| 与本项目匹配度 | ✅ 实时曲线场景 | ⚠️ 静态报表场景 |

**结论**：选择 XChart，理由为实时更新 API 更简洁，更适合 800ms 动态刷新场景。

**（3）ORM 方案对比**

| 维度 | 自研 BaseDao<T> | Hibernate | MyBatis |
|------|----------------|-----------|---------|
| 学习成本 | 反射基础即可 | 高（JPA/HQL/缓存） | 中（XML/注解） |

#### 4.2.2 组员1 技术实践（核心引擎模块）

**负责范围**：`src/main/java/com/iso11820/core/`（8 个 Java 文件，3,026 行）

**关键技术实践**：

**（1）TestStateMachine — 线程安全有限状态机**

- **技术决策**：使用 `AtomicReference<TestState>` 而非 `synchronized` 方法，减少锁竞争。状态切换的 check-and-set 原子性由 `synchronized(this)` 锁保证，监听器通知在锁外执行避免死锁。
- **技术细节**：`canTransitionTo(target)` 使用 Java 17 的 `switch` 表达式（而非传统的 `switch-case`），每个状态枚举值声明其允许的目标状态集合，状态流转规则自包含在 `TestState` 枚举中。
- **测试验证**：组员5 编写 35+ 用例覆盖全部 25 个状态对的合法/非法切换、4 个强制切换场景、6 个监听器场景、3 个并发安全场景。

**（2）SensorSimulator — 多阶段温度仿真引擎**

- **技术决策**：使用 `synchronized` 方法保证每次 `update()` 迭代的原子性，避免多线程并发调用导致温度数据不一致。随机数生成器使用固定 `Random(seed)` 确保可重现性。
- **技术细节**：
  - 升温阶段：`tf1 += heatingRate × 0.8 + noise`，0.8 系数对应 800ms 迭代间隔
  - 稳定阶段：`tf1 = targetTemp + noise`，独立噪声通道确保 tf1 ≠ tf2
  - 记录阶段：表面温 `ts += (target - ts) × 0.02`（一阶指数平滑，系数 0.02），中心温 `tc += (target - tc) × 0.01`（系数 0.01，比表面温慢一半）
  - 降温阶段：`tf1 -= 0.5 + noise × 0.1`，钳位到不低于初始温度
- **测试验证**：组员5 编写 16 用例覆盖四阶段算法、噪声范围、稳定判定、降温钳位、线程安全。

**（3）TestMaster — 试验主调度控制器**

- **技术决策**：使用 `ScheduledExecutorService`（单线程守护线程）以 800ms 固定间隔调度 `onTick()`。`tickLock` 锁保证控制方法（startHeating/stopRecording 等）与 `onTick()` 之间的互斥。
- **技术细节**：
  - `handlePreparingTick()` 检测 `simulator.isStable()` 自动切换到 READY
  - `handleReadyTick()` 持续采样功率值，调用 `checkAndHandleReadiness(tf1)` 检测温度回退
  - `handleRecordingTick()` 每秒追加一条数据到 `CopyOnWriteArrayList`，增量更新最大值/最小值/累加器，检测终止条件（到达时长 / 温漂达标）
  - `calcTempDrift()` 使用最小二乘法对最近 10 分钟（600 个数据点）的炉温序列做线性回归，O(n) 等差数列求和公式优化
- **测试验证**：组员5 编写 20+ 用例覆盖完整生命周期、非法操作拦截、保存状态保护、统计验证、边界值、并发安全。

#### 4.2.3 组员2 技术实践（数据持久层模块）

**负责范围**：`src/main/java/com/iso11820/entity/` + `dao/` + `sql/`（20+ 个 Java 文件，3,327 行）

**关键技术实践**：

**（1）BaseDao<T> — 泛型 CRUD 反射映射**

- **技术决策**：通过反射解析泛型参数 `T` 的 Class 对象，而非要求子类显式传递 Class。使用 `ConcurrentHashMap` 缓存字段映射，首次初始化后零反射开销。
- **技术细节**：
  - `executeQuery(sql, params)` 使用 `PreparedStatement.setObject()` 参数化查询，防止 SQL 注入
  - `mapResultSet(rs)` 遍历 `ResultSetMetaData` 获取列名，通过字段名反射赋值
  - 字段名匹配支持三种命名约定：精确匹配 → 首字母小写（PascalCase → camelCase） → 下划线转驼峰（snake_case → camelCase）
  - 继承场景支持：`findField()` 在实体类及其父类中递归查找字段
- **测试验证**：组员5 编写 7 个 DAO 测试类（35+ 用例），全部使用临时 SQLite 文件隔离。

**（2）DbUtil — 连接管理与版本升级**

- **技术决策**：使用 `ThreadLocal<Connection>` 存储线程绑定连接，批量操作复用同一连接。数据库版本管理通过 `system_config` 表记录 `db_version`，`checkAndUpgrade()` 方法支持从任意版本逐级升级。
- **技术细节**：
  - 首次调用 `getConnection()` 时自动执行 `initDatabase()` → 读取 `schema.sql` → 按分号分割 → 逐条执行。使用 `CREATE TABLE IF NOT EXISTS` 和 `INSERT ... WHERE NOT EXISTS` 保证幂等性
  - 事务接口：`beginTransaction()` 设置 `autoCommit=false` → 业务操作 → `commitTransaction()` 或 `rollbackTransaction()` → `closeTransactionConnection()` 恢复 `autoCommit=true`
  - 资源关闭：`closeResource(AutoCloseable...)` 逐个安全关闭，单个失败不影响后续
