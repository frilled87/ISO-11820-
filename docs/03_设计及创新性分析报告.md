## 1 痛点分析

### 1.1 痛点概述

#### 1.1.1 传统实体不燃性试验设备的核心困境

建筑材料不燃性检测是建筑防火安全评估的第一道关口。ISO 11820:2022 国际标准严格规定了试验流程：将样品置于 750°C 加热炉中，连续监测 60 分钟内的炉内温度、样品表面温度、样品中心温度等多通道数据，依据温升和失重率两项指标判定材料是否合格。然而，当前国内高校和检测机构在实验教学和标准培训中普遍面临四大痛点：

**痛点一：设备采购与维护成本极高**

| 成本项 | 详情 | 金额 |
|--------|------|------|
| 试验炉主机 | ISO 11820 标准不燃性试验炉（含加热元件、炉膛、温控系统） | 15-30 万元 |
| PID 温控模块 | 高精度 PID 控制器 + 电力调节器 | 3-8 万元 |
| Modbus 采集模块 | 多通道温度采集卡 + 传感器组（K 型热电偶 × 5） | 2-5 万元 |
| 串口通信硬件 | RS-485 转换器、数据采集卡、工控机 | 1-3 万元 |
| 年度维护 | 加热元件更换、热电偶校准、设备检定 | 2-5 万元/年 |
| **单台合计** | | **20-50 万元** |

对于高校实验室而言，通常仅能配置 1-2 台设备，而一个教学班 30-40 名学生，人均实操机会极为有限。

**痛点二：实验耗材持续消耗，教学成本叠加**

每次真实试验需要消耗：
- 建筑材料样品（岩棉板、石膏板等）—— 单次 10-50 元
- 热电偶可能因高温损坏 —— 更换成本 200-500 元/支
- 加热炉升温能耗 —— 单次试验约 15-20 kWh

按一个教学班每人完成 1 次实验计算，单次教学实验的耗材成本约 500-1000 元，且样品不可重复使用。

**痛点三：实验数据数字化程度低，教学分析困难**

| 问题 | 影响 |
|------|------|
| 商业软件数据格式封闭 | 无法导出原始数据供学生课后分析 |
| 缺少数据可视化工具 | 学生难以直观理解温度变化趋势 |
| 无标准化报告生成 | 教师需手动整理实验数据，编写实验报告 |
| 历史数据管理缺失 | 学生无法对比不同材料、不同批次的实验结果 |

**痛点四：缺少标准化教学校验与质量保障机制**

真实实验存在以下教学管理难题：
- 学生操作失误（如忘记记录、误触停止）无法"回退"重做
- 实验结果的准确性依赖设备状态，无法提供标准"参考答案"
- 教师无法批量评估学生实验操作规范性
- 缺少自动化评分和操作审计机制

#### 1.1.2 行业背景与政策驱动

| 驱动因素 | 说明 |
|---------|------|
| 标准更新 | ISO 11820:2022 替代旧版标准，试验流程和判定标准均有更新，高校教学大纲需同步调整 |
| 数字化转型 | 教育部《教育信息化 2.0 行动计划》明确要求推进虚拟仿真实验教学 |
| 安全要求 | 750°C 高温试验存在烫伤、火灾风险，教学场景中安全管控压力大 |
| 碳中和目标 | 减少教学实验中的能源消耗和材料浪费，仿真替代是绿色教学方向 |

### 1.2 相关工作

#### 1.2.1 市面同类仿真软件现状与缺陷

当前市场上与 ISO 11820 试验相关的软件产品主要分为三类：商业试验软件、通用仿真平台、高校自研 Demo。我们对三者进行了系统调研和分析：

**（1）商业试验软件（如 Mettler Toledo STARe、NETZSCH Proteus）**

| 维度 | 现状 | 缺陷 |
|------|------|------|
| 功能定位 | 配套特定品牌试验炉的数据采集与控制软件 | 绑定硬件，无独立仿真模式 |
| 教学模式 | 无教学专用功能，面向生产检测 | 无法用于课前预习、课后复现 |
| 数据格式 | 封闭私有格式 | 无法导出 CSV 等通用格式供分析 |
| 许可成本 | 含硬件捆绑销售，单套 20-50 万元 | 高校教学批量部署不可行 |
| 可扩展性 | 闭源，无法二次开发 | 不能根据教学需求定制功能 |

**（2）通用仿真平台（如 MATLAB Simulink、COMSOL）**

| 维度 | 现状 | 缺陷 |
|------|------|------|
| 仿真能力 | 可建立热传导物理模型 | 模型搭建复杂，需传热学专业知识 |
| 操作流程 | 无 ISO 11820 标准流程引导 | 学生无法体验标准试验操作流程 |
| 学习成本 | 需学习 MATLAB/COMSOL 操作 | 增加额外学习负担，偏离教学重点 |
| 报告生成 | 需手动整理数据 | 无标准格式试验报告 |
| 许可成本 | 学校购买正版每年数万元 | 学生个人电脑无法使用 |

**（3）高校自研简易 Demo（如课程设计项目）**

| 维度 | 现状 | 缺陷 |
|------|------|------|
| 功能完整度 | 通常仅实现温度曲线显示 | 缺少完整试验流程、状态管理、数据持久化 |
| 软件质量 | 无单元测试，代码质量参差 | 运行不稳定，容易出现崩溃 |
| 文档规范 | 无标准化文档 | 不可维护、不可传承 |
| 协作模式 | 单人开发为主 | 无团队协作实践，代码结构混乱 |
| 标准符合性 | 算法随意，不遵循 ISO 标准 | 仿真数据不可信，无教学参考价值 |

#### 1.2.2 市场空白总结

综合以上调研，当前市场存在一个明确的**三重空白**：

```
┌─────────────────────────────────────────────────────────┐
│  商业试验软件：太贵（20-50万），绑定硬件，不可教学化      │
│  通用仿真平台：太难（需专业建模），无标准流程引导         │
│  高校自研Demo：太简陋（功能残缺），无质量保障             │
│                                                         │
│  ➤ 市场空白：零成本、教学友好、标准合规、质量可控        │
│    的 ISO 11820 仿真软件 ———— 目前不存在                  │
└─────────────────────────────────────────────────────────┘
```

**本项目的核心设计动机**：正是基于上述痛点分析和市场空白，我们 5 人小组决定开发一套面向高校实验教学场景的 ISO 11820 建筑材料不燃性试验仿真系统，以"零硬件成本、全流程仿真、标准合规、开源可扩展"为设计目标，填补这一市场空白。

## 2 项目创新点

### 2.1 技术性创新点

#### 2.1.1 多阶段温度仿真算法（组员1 负责）

**创新背景**：传统仿真 Demo 通常使用简单的线性插值或随机数生成温度数据，无法反映真实试验炉的热力学特性。ISO 11820 标准中，试验炉的温度变化具有明显的阶段性特征——升温速率、稳定钳位、热传导延迟、降温惯性等。

**创新实现**：组员1 设计并实现了严格遵循 ISO 11820 标准的多阶段温度仿真引擎 `SensorSimulator`，包含四个独立算法阶段：

| 阶段 | 触发条件 | 算法核心 | 数学特征 |
|------|---------|---------|---------|
| **升温阶段** | TF1 < 747°C | `TF1 += HeatingRate × 0.8 + noise` | 线性增长 + 独立噪声 |
| **稳定阶段** | TF1 ≥ 747°C | `TF1 = TargetTemp + noise` | 钳位到目标温度，连续稳定计数 |
| **记录阶段** | RECORDING 状态 | `TS += (Target - TS) × 0.02 + noise` | 一阶指数平滑逼近 |
| **降温阶段** | IDLE 状态 | `TF1 -= 0.5 + noise × 0.1` | 线性衰减 + 钳位下限 |

**算法创新点**：

1. **独立噪声通道**：炉温1 和炉温2 使用独立随机噪声生成，模拟真实试验炉中两支热电偶的测量差异，而非简单复制同一数值
2. **指数逼近热传导模型**：表面温向炉温×0.95 指数接近（系数 0.02），中心温向炉温×0.85 指数接近（系数 0.01），模拟真实热传导中表面升温快、中心升温慢的物理特性
3. **可配置参数体系**：目标温度、升温速率、噪声系数、稳定阈值全部从配置文件读取，支持教师根据教学需求调整仿真难度（如加快升温速度用于课堂演示，或使用真实速率用于模拟考试）
4. **固定种子可重现**：仿真引擎支持指定随机种子，确保同一配置下多次运行产生完全一致的仿真数据，为教学提供"标准答案"参考

**协作关系**：该仿真引擎通过 `DataChangeListener` 事件接口向 UI 层（组员3）推送温度数据，通过 `TestMaster` 主控制器（组员1）与状态机联动，试验完成后数据经 DAO 层（组员2）持久化，最终由导出服务（组员4）生成报告。全流程经组员5 编写的 `SensorSimulatorTest`（16 个测试用例）验证正确性。

#### 2.1.2 线程安全有限状态机（组员1 负责）

**创新背景**：ISO 11820 试验流程包含 5 个明确的状态（IDLE → PREPARING → READY → RECORDING → COMPLETE），状态间存在严格的流转规则和自动判定条件。传统实现通常使用 `if-else` 或 `switch-case` 硬编码状态逻辑，扩展性差且容易出错。

**创新实现**：组员1 设计并实现了基于 `AtomicReference` 的线程安全有限状态机 `TestStateMachine`，包含以下创新设计：

| 设计要素 | 实现 | 创新点 |
|---------|------|--------|
| 状态定义 | `TestState` 枚举，每个状态声明 `canTransitionTo(target)` 规则 | 状态流转规则自包含在枚举中，新增状态只需修改枚举定义 |
| 线程安全 | `AtomicReference<TestState>` + `synchronized` 锁 | check-and-set 原子性保证，避免 ABA 问题 |
| 监听器模式 | `Consumer<StateChangeEvent>` 回调 + 不可变副本遍历 | 监听器在锁外执行，避免死锁；单个监听器异常不影响其他 |
| 自动回退 | `checkAndHandleReadiness(tf1)` | READY 状态下温度跌出 745~755°C 范围自动回退 PREPARING |
| 强制切换 | `forceTransitionTo(target)` | 跳过校验的紧急切换，用于系统重置场景 |
| 事件记录 | `StateChangeEvent` record 类型（Java 17） | 不可变事件对象，含 oldState/newState/timestamp，支持 isForward()/isRollback() 判定 |

**协作关系**：状态机由 `TestMaster` 主控制器（组员1）调用，状态变更通过 `StateChangeEvent` 通知 UI 层（组员3）更新按钮状态和系统消息。组员5 编写的 `TestStateMachineTest`（35+ 测试用例）覆盖了全部合法切换路径、非法拦截、监听器通知、强制切换、温度回退和并发安全场景。

#### 2.1.3 分层持久化事务架构（组员2 负责）

**创新背景**：传统课程设计项目通常直接在代码中拼接 SQL 字符串，存在 SQL 注入风险、代码重复严重、字段映射易出错、无事务保护等问题。

**创新实现**：组员2 设计并实现了一套完整的分层数据持久化架构：

| 层次 | 组件 | 创新点 |
|------|------|--------|
| **连接管理层** | `DbUtil` | 首次运行自动建表、ThreadLocal 连接绑定（批量复用）、事务 begin/commit/rollback 封装、数据库版本管理（`system_config` 表支持升级迁移） |
| **泛型 CRUD 层** | `BaseDao<T>` | 反射解析泛型参数自动推断实体类型、PreparedStatement 参数化查询防注入、字段缓存（ConcurrentHashMap）首次初始化后零反射开销、三种命名转换（精确匹配/首字母小写/下划线转驼峰） |
| **接口抽象层** | 6 个 DAO 接口 | `OperatorDao`、`TestMasterDao`、`ProductMasterDao`、`ApparatusDao`、`SensorsDao`、`CalibrationRecordsDao`，面向接口编程，便于替换实现 |
| **实体映射层** | 6 个 Entity 类 | 纯 POJO，字段与数据库列名对应，与 BaseDao 反射映射协同 |

**架构创新点**：

1. **泛型 CRUD 消除重复代码**：传统 DAO 开发中，每个实体类都需要编写几乎相同的 `insert/update/delete/select` 代码。`BaseDao<T>` 通过反射一次性实现通用 CRUD，6 个 DAO 实现类仅需编写业务特有查询方法，代码量减少 60%+
2. **测试友好设计**：`DbUtil.setDbPath()` 支持动态切换数据库路径，配合 `BaseDaoTest` 基类的 `@BeforeEach` 创建临时 SQLite 文件、`@AfterEach` 自动删除，实现单元测试与正式数据完全隔离
3. **版本化升级策略**：`system_config` 表记录 `db_version`，`checkAndUpgrade()` 方法支持从任意版本逐级升级到目标版本，后续新增字段或表时只需添加对应的 `upgradeTo_X_Y()` 方法

**协作关系**：DAO 层为 Core 层（组员1）和 UI 层（组员3）提供数据访问服务，为导出服务（组员4）提供试验数据读取。组员5 编写的 7 个 DAO 测试类（35+ 用例）全部通过临时数据库隔离测试。

#### 2.1.4 自动化质量门禁体系（组员5 负责）

**创新背景**：传统课程设计项目通常缺少自动化测试，代码质量无法保证，"跑通就行"成为常态。学生提交的项目往往存在未发现的边界 Bug，评审老师也难以量化评估代码质量。

**创新实现**：组员5 牵头搭建了一套完整的自动化测试与质量门禁体系：

| 组件 | 工具 | 创新点 |
|------|------|--------|
| 单元测试框架 | JUnit 5.10 | `@Nested` 分层组织测试用例、`@ParameterizedTest` 参数化测试、`@DisplayName` 中文命名 |
| 覆盖率引擎 | JaCoCo 0.8.12 | Maven `verify` 阶段自动检查，INSTRUCTION 覆盖率 < 80% 构建失败 |
| 测试隔离 | 临时 SQLite + 固定种子 | 数据库测试使用 `Files.createTempFile()`，仿真测试使用固定 `Random(42)` |
| 测试基类 | `BaseDaoTest` | `@BeforeEach` 自动建临时库 + `@AfterEach` 自动删除，子类零配置 |

**质量门禁流程**：

```
组员1-4 提交代码 → 组员5 编写/更新对应测试
        ↓
  mvn clean verify（自动化）
        ↓
  JaCoCo 覆盖率检查 → < 80% → 构建失败 → 通知补充测试
        ↓ ≥ 80%
  生成覆盖率报告 → 提交通过
```

**创新价值**：该体系不仅保障了本项目代码质量，更重要的是为团队协作开发提供了一套可复制的质量保障范式——每位组员提交的代码必须通过自动化测试验证，避免了"我的代码在我电脑上能跑"的协作困境。

### 2.2 功能性创新点

#### 2.2.1 可视化实时多通道温度曲线与全流程交互（组员3 负责）

**创新背景**：商业试验软件的操作界面通常面向生产检测人员设计，操作流程复杂、学习曲线陡峭，不适合教学场景。学生需要的是一个操作流程清晰、反馈直观、状态可感知的交互界面。

**创新实现**：组员3 设计并实现了基于 JavaFX 21 + FXML 的完整用户界面，包含以下功能创新：

**（1）实时多通道温度曲线**

| 特性 | 实现 | 创新点 |
|------|------|--------|
| 4 线折线图 | XChart 组件，炉温1（红）、炉温2（蓝）、表面温（绿）、中心温（橙） | 颜色区分直观，4 通道同步对比 |
| 滚动窗口 | 最近 10 分钟数据，X 轴自动滚动 | 聚焦当前试验状态，不显示过期数据 |
| 动态更新 | 每 800ms 追加数据点，`Platform.runLater()` 线程安全刷新 | 实时性 + 稳定性兼顾 |
| 嵌入方式 | SwingNode 嵌入 JavaFX 场景 | 解决 XChart（Swing 组件）与 JavaFX 的兼容性问题 |

**（2）LED 风格数值面板**

5 通道温度值以大字体独立显示，保留 1 位小数，模拟真实试验设备上的数字仪表盘，增强操作沉浸感。

**（3）按钮状态智能联动**

| 当前状态 | 开始升温 | 停止升温 | 开始记录 | 停止记录 | 新建试验 |
|---------|:------:|:------:|:------:|:------:|:------:|
| IDLE | ✅ | ❌ | ❌ | ❌ | ✅ |
| PREPARING | ❌ | ✅ | ❌ | ❌ | ❌ |
| READY | ❌ | ✅ | ✅ | ❌ | ❌ |
| RECORDING | ❌ | ❌ | ❌ | ✅ | ❌ |
| COMPLETE | ❌ | ✅ | ❌ | ❌ | 未保存❌ |

按钮状态严格跟随状态机流转，从 UI 层面杜绝用户误操作。

**（4）系统消息彩色日志**

状态切换、温度稳定、异常回退等关键事件自动生成带时间戳的系统消息，不同事件类型显示不同颜色（普通白色、警告黄色），模拟真实试验设备的操作日志。

**协作关系**：UI 层通过 `DataChangeListener` 接收 Core 层（组员1）推送的温度数据和状态变更，通过 DAO 层（组员2）查询历史记录，调用导出服务（组员4）生成报告。组员5 对 UI 交互流程进行手动集成测试验证。

#### 2.2.2 多格式自动报告导出（组员4 负责）

**创新背景**：传统实验教学中，学生需要手动记录数据、截图、整理报告，耗时且容易出错。商业软件通常仅支持单一格式导出，无法满足教学场景中"原始数据（CSV）+ 分析报告（Excel）+ 正式文档（PDF）"的多样化需求。

**创新实现**：组员4 设计并实现了三格式自动报告导出体系：

**（1）CSV 原始数据导出**

| 特性 | 实现 |
|------|------|
| 编码 | UTF-8 + BOM（Excel 直接打开无乱码） |
| 表头 | `Time,Temp1,Temp2,TempSurface,TempCenter,TempCalibration,AmbientTemp,AmbientHumidity,Timestamp` |
| 写入模式 | 批量追加（BufferedWriter + 写锁），减少 IO 次数 |
| 容错 | 文件不存在自动创建目录和表头，格式损坏不崩溃 |

**（2）Excel 三 Sheet 报告**

| Sheet | 内容 | 技术 |
|-------|------|------|
| Sheet1：试验信息 | 样品编号、试验日期、操作员、环境参数、质量数据、温升、失重率、判定结论 | Apache POI 格式化表格 |
| Sheet2：温度数据 | 全量每秒温度数据表，含表头样式和自动列宽 | Apache POI 数据填充 |
| Sheet3：温度曲线 | 炉温1/炉温2/表面温/中心温 四线折线图 | Apache POI ChartSheet + XChart 生成图片嵌入 |

**（3）PDF 正式报告**

| 特性 | 实现 |
|------|------|
| 页面 | A4，含页眉页脚页码 |
| 内容 | 试验概要（基本信息表 + 判定结论）+ 温度曲线图 + 判定依据说明 |
| 中文渲染 | iText 7 pdfCalligraph 模块，支持中文字体正确渲染 |
| 排版 | 表格 + 图片 + 文字混排，符合正式报告格式 |

**（4）导出门面模式**

`ReportExportService` 封装了 Excel 和 PDF 两种导出服务，对外仅暴露一个 `exportReport(ExportTestInfo, callback)` 方法，内部根据 `appsettings.json` 配置自动判断导出格式组合，支持导出进度回调。

**协作关系**：导出服务读取 Core 层（组员1）的试验统计数据、DAO 层（组员2）的数据库记录、CSV 数据文件（组员4 的 CsvDataService），最终生成报告文件。组员5 编写的工具类测试覆盖了导出相关的 NumUtil、DateUtil、JsonUtil 等基础组件。

### 2.3 团队协作创新点

#### 2.3.1 Git 模块化分目录协同开发

**创新背景**：传统课程设计团队项目中，组员通常在同一目录下混合编码，频繁出现 Git 合并冲突，且无法精确统计每位成员的代码贡献。代码耦合度高，一人修改导致他人代码不可用的"雪崩效应"时有发生。

**创新实践**：本小组采用**模块化分目录独立开发**模式，5 位组员各自拥有独立的 Git 工作目录：

```
Git 仓库根目录
├── src/main/java/com/iso11820/core/        ← 组员1 独占
├── src/main/java/com/iso11820/entity/      ← 组员2 独占
├── src/main/java/com/iso11820/dao/         ← 组员2 独占
├── sql/                                    ← 组员2 独占
├── src/main/java/com/iso11820/ui/          ← 组员3 独占
├── src/main/resources/fxml/                ← 组员3 独占
├── src/main/java/com/project/service/      ← 组员4 独占
├── src/main/java/com/project/utils/        ← 组员4 独占
├── src/test/java/                          ← 组员5 独占
└── docs/                                   ← 组员5 独占
```

**创新价值**：

| 维度 | 传统混合开发 | 本小组模块化开发 |
|------|------------|----------------|
| Git 合并冲突 | 频繁，每次合并需手动解决 | **零冲突**（8 次提交，0 次冲突） |
| 代码贡献追溯 | 无法精确统计 | `git log -- <directory>` 精确到行 |
| 并行开发效率 | 互相阻塞 | 5 人同时开发，互不干扰 |
| 模块边界 | 模糊，耦合严重 | 接口清晰，依赖单向 |
| 代码审查 | 难以定位责任人 | 直接查看目录所有者 |

**协作机制**：
- 接口先定：开发前组员共同商定模块间接口（`DataChangeListener`、DAO 接口、`ExportTestInfo` 等），接口一旦确定不再修改
- 独立开发：各模块内部实现完全自主，组员之间互不干预
- 集成联调：各模块开发完成后，通过接口进行集成，问题定位到具体模块

#### 2.3.2 全流程 AI 辅助开发（Claude Code）

**创新背景**：AI 辅助编程是当前软件工程领域的热门方向，但在高校课程设计项目中，学生团队如何系统性地将 AI 工具融入开发流程、提升开发效率和代码质量，尚无成熟的实践范式。

| Bug 修复 | 粘贴错误日志 → Claude Code 分析根因 → 提供修复方案 | 调试时间缩短 50%+ |
| 文档撰写 | 提供模板 + 项目信息 → Claude Code 生成初稿 → 全员审核 | 5 份文档快速产出 |
| 代码审查 | 提交 Diff → Claude Code 审查 → 发现问题 → 修复 | 代码质量提升 |

**创新价值**：

#### 2.3.3 标准化五套竞赛文档交付规范

**创新背景**：大赛要求提交 5 份独立文档，传统做法是各组员分头撰写，导致文档格式不统一、内容重复冗余、编号混乱。评审老师在不同文档间切换时体验割裂。

**创新实践**：组员5 牵头制定了统一的文档格式规范，5 份文档共享：

| 规范项 | 统一标准 |
|--------|---------|

| 排版规范 | 一级标题 16pt 加粗、二级 14pt 加粗、正文 12pt 1.5 倍行距 |
| 内容策略 | 五份文档各有侧重，不重复不遗漏 |

**内容差异化分工**：

| 文档 | 核心问题 | 目标读者 |
|------|---------|---------|
| 项目计划书 | 做什么、谁来做、花多少 | 管理评委 |
| 项目开发文档 | 怎么做、怎么设计、怎么实现 | 开发评委 |
| 项目测试文档 | 怎么验证质量、测试了什么 | 测试评委 |
| 技术研究报告 | 解决什么技术问题、怎么验证 | 技术评委 |
| 设计及创新性分析报告 | 为什么比别人好、创新在哪 | 创新评委 |

**创新价值**：该规范体系使 5 份文档形成有机整体，而非零散堆砌，提升了评审体验和项目专业度。
