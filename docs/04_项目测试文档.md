## 1 测试计划

### 1.1 测试策略与目标

#### 1.1.1 测试策略

本项目的测试策略在软件需求分析完成后即开始实施，采用**分层测试 + 自动化门禁 + 人工功能验证**三位一体的质量保障体系：

| 测试层级 | 策略 | 执行方式 | 负责方 |
|---------|------|---------|--------|
| **单元测试** | 对核心引擎、DAO 层、工具类全部公开方法编写 JUnit 5 单元测试，使用内存 SQLite 临时数据库隔离测试环境 | 自动化（`mvn test`） | 组员5 统筹编写，组员1/2/4 协同联调 |
| **覆盖率门禁** | JaCoCo 0.8.12 配置 INSTRUCTION ≥ 80% 覆盖率门禁，`mvn verify` 阶段自动检查，不达标构建失败 | 自动化（`mvn verify`） | 组员5 配置 |
| **功能测试** | 对 6 大业务流程进行端到端人工功能验证，覆盖正常路径和异常路径 | 手动测试 | 组员3 提供 UI 操作支持，组员5 执行用例并记录结果 |
| **性能测试** | 对曲线渲染、文件读写、报告导出进行性能基准测试 | 手动 + 工具辅助 | 组员5 执行 |

**测试重点**：核心引擎状态机流转正确性、温度仿真算法数学精度、DAO 层 CRUD 正确性与事务一致性、工具类边界条件处理。

**测试难点**：温度仿真算法的随机噪声验证需要大量采样（1000+ 次迭代）、状态机并发安全的可重现验证、SQLite 测试隔离的环境自动管理。

**测试分层**：

```
┌──────────────────────────────────────────────┐
│              系统测试（性能 + 覆盖率）          │
│         全系统联调，验证性能指标和覆盖率目标      │
├──────────────────────────────────────────────┤
│              功能测试（6 大业务流程）            │
│   登录 → 新建试验 → 升温 → 记录 → 导出 → 查询   │
├──────────────────────────────────────────────┤
│              单元测试（3 大模块）               │
│  核心引擎(55+用例) │ DAO层(35+用例) │ 工具类(85+用例) │
└──────────────────────────────────────────────┘
```

1. **状态机测试必须覆盖"非法切换"路径**：正常流程测试只能验证"程序能做正确的事"，非法切换拦截测试才能验证"程序不会做错误的事"。我们在测试中发现了 RECORDING→IDLE 的非法跳转未被拦截的 Bug，正是因为专门编写了非法切换测试用例。
2. **并发测试需要多轮重复验证**：并发 Bug 具有偶发性，`@RepeatedTest(3)` 多次重复执行才能提高发现概率。
3. **固定种子是可重现性的关键**：`SensorSimulator(42)` 确保每次测试产生完全相同的温度序列，使测试结果可被任何开发者复现。

#### 4.1.2 测试结果综合分析及建议

全部 8 项性能指标均达到或超过预期目标。其中：

- **曲线渲染**：XChart 的 800ms 刷新在 600 数据点窗口下表现稳定，未出现掉帧
- **文件导出**：CSV（批量 BufferedWriter）和 Excel（Apache POI Streaming）均达到预期性能，PDF（iText 7）因中文渲染开销略高但在可接受范围内
- **内存占用**：得益于 CopyOnWriteArrayList 的只读快照和 ScheduledExecutorService 的单线程调度，内存占用控制良好

建议后续增加：
- 大数据量场景（10000+ 行）的导出性能测试
- 长时间运行（> 2 小时）的内存泄漏检测
- 低配置机器（4GB RAM）上的性能基准测试

#### 4.1.3 测试经验总结

性能测试的关键是**建立可量化的基准指标**并在每次发布前验证。我们为每个性能敏感操作设定了明确的阈值（如曲线渲染 < 100ms、Excel 导出 < 5s），并在功能测试中嵌入性能监控点。
